# プロダクト要求定義書

## 1. プロダクトビジョンと目的

### ビジョン

開発者がタスク管理の認知コストをゼロにし、コードを書くことだけに集中できる環境を作る。
Gitコンテキストと連携した自動優先順位調整により、「次に何をやるか」を考える時間を排除する。

### 目的

開発者がコードと向き合う時間を最大化するために、タスクの優先順位をコンテキストに応じて自動調整し、ターミナルから離れることなくタスク管理を完結させる。

### 解決する課題

- 既存のタスク管理ツール（Todoist、Trello、Asana）はコードの文脈と切り離されており、優先度はユーザーが手動で設定・更新しなければならない
- GUIベースのツールはターミナルから開発者を引き離し、コンテキストスイッチを発生させる
- 「次に何をやるべきか」を判断する認知コストが高く、集中を妨げる

## 2. ターゲットユーザーと課題・ニーズ

### ターゲットユーザー

**対象：** ターミナルを主な作業環境とし、Gitを日常的に使用する個人開発者（主要ターゲット）またはスモールチーム（2〜5名）。

**対象外：** 非エンジニア、GUI前提のチーム、大規模プロジェクト管理が必要な組織（50名以上）。

> スモールチームを対象に含めるが、v1ではタスクの共有・アサイン機能は提供しない。
> チーム向けは将来バージョンで検討する。

### ペルソナ

**山田さん（32歳、フリーランスエンジニア）**

- 受託案件を3〜4本並行して抱えており、各案件で複数の開発タスクが走っている
- 作業環境はVSCode + ターミナル中心。ブラウザを開くのは調査のときだけにしたい
- 現在はGitHub Issuesとメモ帳を併用しているが、コンテキストが分散している
- 毎朝「今日何から手をつけるか」を考えるのに10〜15分かかっており、それをなくしたい
- 締め切りを手帳で管理しているが、タスクと紐付いていないため見落とすことがある

### 課題

| 課題 | 詳細 |
|------|------|
| コンテキストスイッチのコスト | タスク管理のためにブラウザ・GUIを開く手間と集中の断絶 |
| 優先度の陳腐化 | 状況が変わっても優先度を手動で更新しないと古いまま残る |
| コードとタスクの乖離 | どのブランチ・ファイルに関連するタスクかが不明 |
| 「次にやること」の判断疲れ | タスクが増えると何から手をつけるか迷う時間が生まれる |

### ニーズ

- ターミナルだけでタスクの追加・確認・完了ができる
- 今の作業コンテキスト（ブランチ、締め切り、依存関係）に基づいて優先度が自動的に変わる
- 「次にやるべきタスク」を一瞬で把握できる

## 3. 主要機能一覧

| # | 機能 | 概要 |
|---|------|------|
| 1 | タスクCRUD | 作成・参照・更新・削除 |
| 2 | 自動スコアリング | 複数シグナルからスコアを算出し優先順位を自動調整 |
| 3 | `task next` コマンド | スコア最上位のタスクを1件表示し次のアクションを提示 |
| 4 | Gitコンテキスト連携 | 現在のブランチとタスクを自動紐付け |
| 5 | TODO同期 | コード内の `TODO:` コメントをタスクとして取り込む |
| 6 | 依存関係管理 | タスク間のブロック依存を定義し優先度に反映 |
| 7 | スヌーズ・ピン | 一時的な除外と常時上位固定 |
| 8 | プリセット設定 | `deadline` / `balanced` / `flow` から重み付けを選択 |

## 4. 成功の定義

### 定量指標（システムで計測可能）

| 指標 | 目標値 | 計測方法 |
|------|--------|---------|
| `task next` の1日平均実行回数 | 5回以上 | コマンド実行ログ（opt-in） |
| `task next` で提示されたタスクが `done` に繋がる率 | 60%以上 | next → done の遷移をトラッキング |
| 全コマンドの応答時間 | 200ms以内 | 実行時計測 |
| インストール後7日継続利用率 | 70%以上 | 初回実行日から7日後の利用有無 |

### 定量指標（ユーザーインタビューで計測）

| 指標 | 目標値 | 計測方法 |
|------|--------|---------|
| 「今日何から始めるか」の判断時間 | 現状比50%削減（目安：15分→7分以内） | 導入前後のインタビュー |
| タスク管理のために外部ツールを開く頻度 | 週あたり現状比50%削減 | 導入前後のインタビュー |

### 定性的なゴール状態

- スコアの内訳を `task show` で確認したとき、推薦理由に納得できること
- 優先度を手動で更新する必要をほとんど感じないこと

## 5. ユーザーストーリー

| ID | ストーリー |
|----|-----------|
| US-01 | 開発者として、ターミナルからタスクを追加したい。ブラウザを開かずに済むから。 |
| US-02 | 開発者として、`task next` を打つだけで今やるべきタスクを知りたい。迷う時間をなくしたいから。 |
| US-03 | 開発者として、gitブランチを切ったらそのブランチに関連するタスクが自動的に浮上してほしい。コンテキストを合わせる手間を省きたいから。 |
| US-04 | 開発者として、あるタスクが別のタスクをブロックしていると設定したい。依存関係を考慮した優先度にしてほしいから。 |
| US-05 | 開発者として、締め切りが近づくにつれてタスクが自動的に上位に来てほしい。期限を見落としたくないから。 |
| US-06 | 開発者として、スコアの内訳を確認したい。なぜその優先順位なのか理解して信頼したいから。 |
| US-07 | 開発者として、今日は対応できないタスクを一時的にスヌーズしたい。リストをノイズなく保ちたいから。 |
| US-08 | 開発者として、コード内の `TODO:` コメントをタスクとして取り込みたい。手動で二重登録したくないから。 |

## 6. 受け入れ条件

### AC-01：タスク追加

- `task add "内容"` でタスクが作成される
- `--due`, `--branch`, `--tag` オプションを受け付ける
- `--branch` を省略した場合、現在のgitブランチを自動的にタスクに紐付ける
- gitリポジトリ外で実行した場合はブランチ紐付けをスキップしエラーにしない

### AC-02：`task next`

- スコア最上位のタスクを1件表示する
- スコアの主要要因（締め切り・依存・コンテキスト）を1行で表示する
- 次に打てるコマンド（`done`, `snooze`）をヒントとして表示する

### AC-03：自動スコアリング

- 緊急度・重要度・コンテキスト・経年・疲労の5要素でスコアを算出する
- `task show` でスコアの内訳（各要素の得点）が確認できる
- スコアはタスク一覧表示時にリアルタイムで再計算される

### AC-04：プリセット

- `.taskrc` に `preset = "deadline" | "balanced" | "flow"` を指定できる
- プリセットごとに各シグナルの重み係数が異なる（詳細は機能要件参照）

### AC-05：TODO同期

- `task sync` を実行するとリポジトリ内の `TODO:` コメントを取り込む
- 重複判定はファイルパス + 行番号の組み合わせで行う
- `.gitignore` に記載されたファイル・ディレクトリはスキャン対象外とする

## 7. 機能要件

### データモデル

```
Task {
  id          : number        # 自動採番（1から連番）
  title       : string        # タスク本文
  status      : "todo"
               | "in_progress"
               | "done"
               | "snoozed"
  due_date    : date | null   # 締め切り日（日付のみ、時刻なし）
  branch      : string | null # 紐付けるgitブランチ名
  tags        : string[]      # 任意タグ
  blocks      : number[]      # このタスクが完了待ちにしている他タスクのID
  pinned      : boolean       # ピン固定フラグ
  snoozed_until: date | null  # スヌーズ解除日
  source_file : string | null # TODO同期で取り込んだ場合のファイルパス
  source_line : number | null # TODO同期で取り込んだ場合の行番号
  created_at  : datetime
  updated_at  : datetime
}
```

### スコアリングロジック

```
score = (urgency + importance + context + aging - fatigue) × weight_multiplier
```

#### 各要素の計算

**緊急度（urgency）**

| 締め切りまでの残日数 | 点数 |
|---------------------|------|
| due_date が未設定 | 0 |
| 8日以上 | 0 |
| 4〜7日 | 15 |
| 1〜3日 | 30 |
| 当日（0日） | 40 |
| 超過（負の値） | 40 + 超過日数 × 2（上限なし） |

**重要度（importance）**

| 他タスクからの被依存数 | 点数 |
|----------------------|------|
| 0件 | 0 |
| 1件 | 10 |
| 2件 | 20 |
| 3件以上 | 30 |

**コンテキスト加点（context）**

| 条件 | 点数 |
|------|------|
| 現在のgitブランチとタスクのbranchが一致 | +15 |
| 今日すでに `task next` で表示済み（当日初回除く） | +5 |
| 上記なし | 0 |

※ 複数条件が重なる場合は合算する（上限20点）

**経年加点（aging）**

| 作成からの未着手日数 | 点数 |
|---------------------|------|
| 3日未満 | 0 |
| 3〜6日 | 5 |
| 7〜13日 | 10 |
| 14日以上 | 15 |

**疲労ペナルティ（fatigue）**

| 条件 | 点数 |
|------|------|
| 直前の `task next` 表示タスク | −10 |
| 2回前の `task next` 表示タスク | −5 |
| それ以外 | 0 |

#### プリセット重み係数（weight_multiplier）

プリセットは要素ごとの重み係数として適用する。

| 要素 | deadline | balanced（デフォルト） | flow |
|------|----------|----------------------|------|
| urgency | 2.0 | 1.0 | 0.5 |
| importance | 1.0 | 1.0 | 1.0 |
| context | 0.5 | 1.0 | 2.0 |
| aging | 1.0 | 1.0 | 1.0 |
| fatigue | 1.0 | 1.0 | 1.0 |

各要素のスコアに対して重み係数を乗算した後に合算する。

```
score = urgency × w_urgency
      + importance × w_importance
      + context × w_context
      + aging × w_aging
      - fatigue × w_fatigue
```

pinned = true のタスクはスコア計算をスキップし、常にリスト最上位に表示する。

### プリセット設定

```toml
# .taskrc
[profile]
preset = "balanced"   # deadline | balanced | flow
```

## 8. 非機能要件

| 項目 | 要件 |
|------|------|
| パフォーマンス | 全コマンドの応答時間が200ms以内 |
| データ保存 | ローカルファイル（SQLite）。SaaS不要 |
| オフライン動作 | ネットワーク接続なしで全機能が動作する |
| 移植性 | macOS / Linux で動作する |
| 設定可能性 | `.taskrc` でプリセットおよびスコア重みをカスタマイズできる |
| 出力形式 | `--json` フラグでJSON出力に切り替えられる（スクリプト連携用） |
| データエクスポート | タスクデータをMarkdown / JSON形式でエクスポートできる |
| セキュリティ | データファイルのパーミッションを600（所有者のみ読み書き）に設定する |
| データ整合性 | ファイルロック（flock）により同一ファイルへの同時書き込みを防ぐ |
| バージョンマイグレーション | データファイルにスキーマバージョンを記録し、アップデート時に自動マイグレーションを実行する |
| インストール | Homebrew（macOS）およびバイナリ直接配布（Linux）でインストール可能 |
| エラー表示 | gitリポジトリ外・データ破損・不正オプション時に原因と対処法を明示したメッセージを表示する |
| カラー出力 | `NO_COLOR=1` 環境変数またはオプションでカラーなし出力に切り替えられる |
| タイムゾーン | 締め切り計算はシステムのローカルタイムゾーンを使用する。`.taskrc` で上書き可能 |
| デバッグ | `--verbose` フラグでスコア計算の詳細ログを出力できる |
